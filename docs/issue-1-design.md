# Issue #1 设计文档

## 文档信息

- 主题: 类似 TradingView 的多标的百分比 K 线叠加 Web 页面
- 来源: https://github.com/hangchow/relchart/issues/1
- 状态: Draft
- 日期: 2026-03-06

## 1. 背景

当前仓库尚未进入具体实现阶段，Issue #1 描述了一个从零开始的本地 Web 应用需求：

- 服务启动后，通过 Web URL 传入多个股票代码。
- 程序读取并补齐本地月度 K 线缓存。
- 启动一个本地 Web 页面展示多标的叠加图。
- 图形风格接近 TradingView 日 K，但 Y 轴展示涨跌幅百分比而非价格。
- 时间范围固定，用户不能手动调整。

该需求的核心不是通用行情平台，而是一个单用途的本地可视化工具。

## 2. 目标与非目标

### 2.1 目标

1. 支持启动本地 Web 服务，并通过 URL 查询参数指定股票列表。
2. 支持读取 `data_dir` 下的历史月文件，并在页面/API访问时自动补齐缺失数据。
3. 页面固定展示“前 3 个完整自然月 + 本月至今”的日 K。
4. 多个标的在同一坐标系内叠加显示，Y 轴统一为百分比。
5. 数据存储采用文本文件，格式与 issue 中的样例兼容。
6. 首版支持 US、HK 股票代码，以及 `YF.*` 形式的 Yahoo 原生 symbol 透传，且技术实现共用同一套处理流程。
7. 在首版实现中避免引入 Node 构建链，保持部署简单。

### 2.2 非目标

1. 不做账户体系、权限体系或远程部署设计。
2. 不做分钟级、实时推送或盘口能力。
3. 不做可自由切换时间范围的图表交互。
4. 不做多页面应用。
5. 不在首版中单独为 US、HK 之外的更多市场设计专用前缀；额外标的优先通过 `YF.*` 透传支持。

## 3. 需求澄清与默认决策

Issue 原文存在几处歧义。为了让实现可落地，本文档采用以下默认决策。

### 3.1 时间窗口定义

时间窗口定义为：

- 前 3 个完整自然月；
- 加上当前自然月从 1 号到“最后一个已完成交易日”的数据。

示例：若程序在 `2026-03-06` 启动，则窗口为：

- `2025-12-01` 至 `2025-12-31`
- `2026-01-01` 至 `2026-01-31`
- `2026-02-01` 至 `2026-02-28`
- `2026-03-01` 至 `2026-03-06` 的已完成交易日

注意：这不是滚动 90 天窗口。

### 3.2 “今天”是否包含在当前月数据中

当前月是否包含“今天”的日 K，不以服务器本地日期直接判断，而以标的市场的交易日和数据源返回结果判断：

- 若今天尚未形成最终日线，则不包含今天；
- 若数据源已返回最终日线，则包含今天。

### 3.3 百分比基准

Y 轴使用百分比时，所有 OHLC 值统一按“窗口起始日前一个交易日的收盘价”进行归一化：

`pct = (price / base_close - 1) * 100`

其中：

- `base_close` 为窗口可见首日之前一个交易日的收盘价；
- 若无法取得该值，则回退为窗口可见首日的开盘价。

该规则优先保证首个可见交易日的涨跌语义完整。

### 3.4 复权方式

首版默认使用前复权数据。原因：

- 若窗口内存在拆股、送股等企业行为，原始价格序列会导致百分比图失真；
- 前复权更适合做跨日相对涨跌幅展示。

Yahoo 日 K 接入建议通过 `yfinance` 的自动复权能力获取调整后的 OHLC。若后续切换到不支持前复权的数据源，则需在文档和界面中明确标记为“未复权”。

### 3.5 股票数量上限

多标的 K 线叠加的可读性会随着标的数量快速下降。首版建议限制为最多 5 个标的：

- 1 到 3 个标的可读性较好；
- 4 到 5 个标的仍可接受；
- 超过 5 个标的时大概率出现明显遮挡。

### 3.6 颜色分配

Issue 提到“随机分配颜色”，首版实现采用“按 symbol 稳定映射颜色”的方式：

- 不同股票仍然表现为不同颜色；
- 同一股票在重启后颜色保持一致；
- 比真实随机更利于识别和复现。

### 3.7 市场支持与复用策略

首版支持以下三类输入：

- `US.*`，例如 `US.AAPL`
- `HK.*`
- `YF.*`，例如 `YF.GC=F`

实现原则：

- US、HK 和 `YF.*` 共用同一套 CLI、provider 抽象、存储格式、百分比转换和前端渲染逻辑；
- 不为 HK 单独设计一套模块或单独页面；
- 差异只体现在 symbol 解析、交易日历、市场时区和数据源请求参数；
- 若一次请求中同时包含 US、HK 与 `YF.*`，仍走同一套处理流程，前端按自然日期对齐，某一类标的休市时该 symbol 当天无柱属于预期行为。

## 4. 范围

### 4.1 首版支持范围

1. 命令行启动本地 Web 服务。
2. 支持 US、HK 股票代码，以及 `YF.*` 形式的 Yahoo 原生 symbol。
3. 支持读取和写入月度缓存文件。
4. 支持缺失历史整月补齐。
5. 支持当前月增量补齐。
6. 支持展示叠加的日 K 百分比图。
7. 支持本地页面访问和基础错误展示。

### 4.2 首版暂不支持

1. 用户在页面内新增或删除股票。
2. 任意时间范围缩放、拖拽、切换。
3. 为 US、HK 之外的更多市场设计新的专用输入前缀。
4. 服务端持久任务、异步任务队列。
5. 实时自动刷新。

## 5. 总体架构

系统拆分为 4 层：

1. CLI / Bootstrap 层
2. Data Provider 层
3. Storage / Transform 层
4. Web UI 层

数据流如下：

1. 用户通过命令行启动服务并指定目录参数。
2. 用户在浏览器访问 `/kline?stocks=...` 页面。
3. 前端请求 `/api/chart-data?stocks=...`。
4. 服务端解析股票列表并计算时间窗口和所需月份列表。
5. 程序从 `data_dir` 读取缓存。
6. 对缺失月份或当前月缺失交易日调用数据源补数。
7. 程序将结果按月写回本地文本文件。
8. 程序将原始 OHLC 转换为百分比 OHLC。
9. 服务端返回 JSON 数据，浏览器渲染叠加 K 线图。

## 6. 技术选型

### 6.1 后端

建议使用 Python 3.11+。

推荐选型：

- Web 服务: FastAPI
- ASGI Server: Uvicorn
- 数据模型: dataclass 或 Pydantic
- 文件处理: 标准库

选择理由：

- 仓库当前为空，FastAPI 足够轻且便于输出 JSON；
- 单进程即可同时完成补数和页面服务；
- 便于后续增加健康检查和调试接口。

### 6.2 前端

建议使用静态 HTML + 原生 JavaScript，不引入前端构建链。

图表库推荐：

- Plotly.js

选择理由：

- `opdash` 当前 Web 页面已经在使用 Plotly.js，团队对其交互和视觉风格已有直观认知；
- Plotly.js 原生支持 candlestick trace，足以承载本项目的多标的日 K 叠加需求；
- 在本项目的数据量级下，3 到 5 个标的、约 4 个月日线数据的前端负载可接受；
- 与现有项目复用同一图表组件，比再引入另一套图库更省认知成本。

参考实现：

- 可直接参考 sibling repo `opdash` 的静态前端组织方式；
- 关键参考文件为 `../opdash/web/index.html` 和 `../opdash/web/app.js`；
- `relchart` 可沿用其“FastAPI + 静态页面 + Plotly.react 渲染”的结构，只把 trace 类型从现有 scatter 改为 candlestick，并替换数据模型。

实现方式：

- 将固定版本的 Plotly.js 产物直接放入仓库静态资源目录；
- 后端提供 `index.html` 和 JSON 接口；
- 前端使用 Plotly.js 的 candlestick trace 渲染图表；
- 关闭 rangeslider、modebar、滚轮缩放和拖拽缩放，保证时间窗口固定。

这样可以避免：

- 首版引入 Node/npm；
- 构建链复杂化；
- 运行时依赖外部 CDN。

## 7. 数据源设计

### 7.1 Provider 抽象

定义统一接口：

```python
class DailyBarProvider(Protocol):
    def fetch_daily_bars(
        self,
        symbol: str,
        start_date: date,
        end_date: date,
    ) -> list[DailyBar]:
        ...

    def fetch_previous_close(
        self,
        symbol: str,
        before_date: date,
    ) -> float | None:
        ...

    def fetch_trading_days(
        self,
        symbol: StockSymbol,
        start_date: date,
        end_date: date,
    ) -> list[date]:
        ...
```

`DailyBar` 结构：

```python
@dataclass
class DailyBar:
    symbol: str
    date: date
    open: float
    high: float
    low: float
    close: float
```

补充说明：

- `market` 由 symbol 前缀解析得到，例如 `US.AAPL -> US`，`HK.<code> -> HK`，`YF.GC=F -> YF`；
- 首版 provider 必须至少支持 US、HK 和 `YF.*` 三类输入；
- `YF.*` 的交易日历采用 best-effort 推断，因此 `fetch_trading_days` / `last_completed_trading_day` 应以 `symbol` 为入参，而不是只看 market；
- provider 的其余处理逻辑应与具体输入前缀解耦，避免为 HK 或 `YF.*` 复制一份实现。

### 7.2 首版 Provider 策略

首版建议实现 `YahooProvider`，通过公开 Yahoo Finance 历史行情接口拉取日 K，代码层使用 `yfinance` 作为访问封装。

默认策略：

- 不依赖 Futu OpenD；
- 不要求用户准备行情账号、交易权限或本地行情网关；
- 在页面/API访问时通过互联网请求历史日线数据；
- 日 K 拉取使用 `1d` 周期，优先使用 `yfinance` 的自动复权 OHLC；必要时开启其 repair 逻辑降低异常值风险。

Yahoo symbol 映射规则：

- `US.AAPL -> AAPL`
- `US.BRK.B -> BRK-B`
- `YF.GC=F -> GC=F`
- `YF.SI=F -> SI=F`
- HK input format example intentionally omitted until the final symbol convention is confirmed

实现约束：

- `US.*` 去掉市场前缀后，保留主代码并将 `.` 转为 `-`，以兼容 Yahoo 对美股类别股代码的写法；
- `YF.*` 去掉前缀后原样透传 Yahoo symbol，适用于期货、外汇、指数、加密货币等 Yahoo 原生代码；
- `YF.*` 的交易日历使用 best-effort 推断：首版显式覆盖 `GC=F` / `SI=F`，并按 Yahoo 当前日线表现使用 `XNYS` 风格的完成日判断；对常见 `=X`、`-USD`、`=F`、`^...` 形式也提供默认规则；
- 若 Yahoo 返回 `shortName`，前端标题、图例和 hover 文案优先展示该英文名，同时保留原始 symbol 作为辅助标识；
- `HK.*` 的最终输入格式和到 Yahoo symbol 的映射规则保持独立封装，待输入约定最终确认后再固定；
- 若未来支持更多市场，再在映射层新增 market-specific 规则，不影响上层逻辑。

保留扩展位：

- 后续可新增第二公开数据源作为 fallback；
- 若未来需要切换到带 API Key 的数据源，再增加对应参数。

### 7.3 Provider 失败处理

单个标的补数失败时：

- 若本地已有足够缓存，则允许页面继续启动，并对该标的显示告警；
- 若该标的完全无数据，则该标的不展示，并在页面顶部提示失败原因；
- 若所有标的都失败，则服务启动失败并输出错误日志。

公开数据源补充说明：

- Yahoo / `yfinance` 属于低门槛方案，适合本地工具场景；
- 该方案不依赖 OpenD，但稳定性不如付费或官方授权数据服务；
- 因此缓存层仍然是首版实现的重要组成部分，避免每次启动都完全依赖远端响应质量。

## 8. 存储设计

### 8.1 路径结构

文件树统一定义为：

```text
data_dir/
  us.aapl/
    us.aapl_202602.txt
  us.tsla/
    us.tsla_202602.txt
```

规则：

- 最后一层目录名为小写 symbol；
- 文件名为 `{symbol_lower}_{YYYYMM}.txt`；
- symbol 输入大小写不敏感，但落盘时统一转小写。

### 8.2 文件格式

文件内容：

- 无表头；
- 每行一个交易日；
- 单个空格分隔 5 列；
- 列顺序为 `date open high low close`。

示例：

```text
20260201 260 261 257 260.5
20260202 260.5 263 255 262
```

### 8.3 读写规则

读取时：

- 跳过空行；
- 校验列数必须为 5；
- 校验日期合法；
- 校验价格为正数；
- 出现非法行时记录告警，并跳过该行。

写入时：

- 按日期升序排序；
- 对同一日期去重，后写入的数据覆盖先前数据；
- 使用临时文件写入后原子替换，避免中断造成坏文件。
- 对于非最后一个目标月份，只允许在确认该月份数据完整时写入；
- 若抓到的非最后一个目标月份数据不完整，则直接放弃写入，避免生成“看起来存在、实际不完整”的历史文件。

## 9. 时间窗口与补数策略

### 9.1 月份计算

给定启动日，计算：

1. 前 3 个完整自然月；
2. 当前自然月；

例如在 `2026-03-06` 启动，目标月份列表为：

- `202512`
- `202601`
- `202602`
- `202603`

### 9.2 访问时补数流程

对每个 symbol：

1. 读取目标月份对应文件。
2. 若历史整月文件不存在，则整月抓取；仅在确认该月份数据完整时才写入。
3. 若历史整月文件存在，则默认不回补，只做格式校验。
4. 若当前月文件不存在，则抓取本月 1 号到最新已完成交易日。
5. 若当前月文件存在，则计算应有交易日集合并补齐缺失日期。
6. 完成后重新聚合窗口内所有数据。

补充约束：

- “最后一个目标月份”通常指当前月，它允许天然不完整；
- 非最后一个目标月份在业务语义上必须是完整月；
- 若提供方返回了不完整的历史整月数据，则该月本次视为抓取失败，不生成文件。
- 以上流程在页面/API访问该 symbol 时触发，而不是在服务启动时触发。

### 9.3 为什么当前月要“按交易日检查”而不是只向后追加

Issue 原文更接近“往后补充”，但只按最后一天追加有风险：

- 若中间某天因中断未写入，会永久缺洞；
- 若提供方延迟回补某天数据，也无法修复。

因此本文档采用“当前月按交易日集合补齐”的规则。

### 9.4 历史月份修复策略

默认行为：

- 历史月份文件存在即视为冻结，不主动重抓。
- 若后续极少数情况下确认某历史月文件确实缺数据，人工删除对应文件后，程序下次访问相关 symbols 页面/API 时会重新抓取并生成。

可选扩展：

- 增加 `--repair-history` 参数；
- 指定后重新拉取窗口内所有月份并覆盖写回。

## 10. 百分比转换与序列对齐

### 10.1 转换规则

对每个 symbol，先求 `base_close`，再对每根 K 线进行转换：

```python
pct_open = (open / base_close - 1) * 100
pct_high = (high / base_close - 1) * 100
pct_low = (low / base_close - 1) * 100
pct_close = (close / base_close - 1) * 100
```

### 10.2 日期对齐

前端不强制所有 symbol 补出完全相同的日期集合：

- 每个 symbol 仅传自身真实存在的交易日数据；
- 图表库按日期对齐显示；
- 对于某个日期缺失的 symbol，图表表现为该 symbol 当日无柱。

### 10.3 多市场处理

首版直接支持 US、HK 和 `YF.*`，且使用同一套实现。

处理原则：

- 每个 symbol 按自身输入前缀和推断后的交易日历获取日线数据；
- 百分比转换逻辑对 US、HK 和 `YF.*` 完全一致；
- 前端统一按日期轴渲染，不强行补出另一市场不存在的交易日；
- 若一次请求中同时包含 US、HK 与 `YF.*`，不同交易日历造成的日期缺口按“无该日数据”处理。

## 11. Web 设计

### 11.1 页面组成

页面采用单屏布局：

1. 顶部标题区
2. 标的图例区
3. 主图表区域
4. 错误与数据状态提示区

### 11.2 交互策略

允许：

- 十字线查看数值；
- hover 查看各标的当日数据；
- 浏览器窗口缩放后的自适应重绘。

禁用：

- 鼠标滚轮缩放时间范围；
- 拖拽平移时间范围；
- 手势缩放；
- 页面内切换时间周期；
- Plotly 默认的 rangeslider 和 modebar 金融交互入口。

### 11.3 视觉策略

为降低遮挡：

- 使用较细 K 线边框和 wick；
- candlestick body 适当增加透明度；
- 每个 symbol 使用不同主色；
- 使用固定图例标识 symbol 与颜色。

Plotly 渲染策略：

- 每个 symbol 对应一条 candlestick trace；
- 所有 trace 共用同一条日期轴和百分比 Y 轴；
- 使用自定义 hover 内容展示 symbol、日期、百分比 OHLC；
- 默认隐藏 Plotly 工具栏，避免暴露与本需求冲突的自由缩放能力。

## 12. HTTP 接口

### 12.1 页面接口

- `GET /`
  - 返回主页面壳子；不触发数据抓取

- `GET /kline?stocks=...`
  - 返回指定股票列表的页面壳子，例如 `/kline?stocks=US.AAPL,US.TSLA`
  - 页面加载后再请求数据接口并触发抓取/修复

### 12.2 数据接口

- `GET /api/chart-data?stocks=...`
  - 返回页面所需的全部数据
  - 访问该接口时触发对应股票列表的数据抓取和缓存修复

示例响应：

```json
{
  "window": {
    "start": "2025-12-01",
    "end": "2026-03-06"
  },
  "series": [
    {
      "symbol": "US.AAPL",
      "color": "#0EA5E9",
      "base_close": 244.52,
      "bars": [
        {
          "time": "2025-12-01",
```

## 13. CLI 设计

### 13.1 启动命令

推荐首版命令：

```bash
python relchart.py
```

### 13.2 参数定义

- `--data_dir` 默认 `./.stocks`
- `--web_host` 默认 `127.0.0.1`
- `--web_port` 默认 `19090`
- `--repair-history` 可选，默认关闭

访问示例：

```text
http://127.0.0.1:19090/kline?stocks=US.AAPL,US.TSLA
```

Yahoo 原生 symbol 示例：

```text
http://127.0.0.1:19090/kline?stocks=YF.GC%3DF,YF.SI%3DF
```

### 13.3 参数与请求校验

- `data_dir` 若不存在则自动创建
- 端口必须为合法范围
- URL 查询参数 `stocks` 不能为空
- `stocks` 中的股票数量不得超过 5

## 14. 日志与错误处理

建议日志分三类：

- 启动日志
- 数据补齐日志
- Web 请求日志

关键日志内容：

- 窗口起止日期
- 每个 symbol 缓存命中情况
- 每个月份是否触发补数
- 补数成功条数
- 每次本地文件读取的单次耗时
- 每次远程请求的单次耗时
- 单个页面请求维度的本地读取总耗时与远程请求总耗时
- 非法文件行数
- 页面启动地址

4. 多标的颜色稳定不变

## 17. 风险与待确认项

1. Yahoo / `yfinance` 是公开且低门槛的数据源，但并非官方稳定 SLA 服务；若未来出现限流、字段变动或短时不可用，需要准备 fallback 方案。
2. 当前月是否允许展示“当日尚未收盘的临时日 K”。本文档默认不允许。
3. 历史月份默认冻结，仅在人工确认文件缺失或损坏后，通过删文件并重启程序的方式触发重抓。
4. `YF.*` 的交易日历推断属于 best-effort；若后续要覆盖更多 Yahoo 原生代码，可能需要继续补充明确映射规则。
5. 是否接受最多 5 个标的的首版限制。

## 18. 实施顺序建议

1. 先实现 CLI、时间窗口、文件读写和伪 provider。
2. 再实现真实 provider 与补数逻辑。
3. 然后输出 `/api/chart-data?stocks=...`。
4. 最后接入静态页面和图表库。
5. 完成后补齐测试和示例数据。

## 19. 结论

这个需求适合做成一个单进程、本地优先、无前端构建链的小型工具。实现重点不在页面壳子，而在以下 4 个地方：

1. 时间窗口语义必须固定；
2. 月度缓存补数必须稳定；
3. 百分比基准必须统一；
4. 多标的叠加图的交互必须尽量收敛。

只要以上 4 点先定清，后续实现成本和返工风险都会明显下降。
